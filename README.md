# Viewshed API — расчёт зоны видимости станции

Сервис рассчитывает зону доступной к обзору местности для станции
объективного контроля по матрице рельефа и возвращает результат в
формате **GeoJSON**, а также в виде PNG-визуализации.

## Что делает алгоритм

Для каждой ячейки в радиусе `r` от станции трассируется прямой луч.
Ячейка считается видимой, если ни одна точка рельефа на пути луча
не перекрывает прямую видимость до неё. Строго:

```
visible(target)  ⇔  angle(target) ≥ max angle(rельеф_i)
                                    i ∈ путь
где angle(p) = (h_terrain(p) − (h_terrain(station) + h_station)) / dist(station, p)
```

Каждая видимая ячейка соответствует **точке** в узле сетки
`(j·cell_size, i·cell_size)`. Все такие точки собираются в
GeoJSON-геометрию `MultiPoint` — без объединения в полигоны
(алгоритм оценивает видимость точечно, а не площадно).

## Установка

Требуется [uv](https://docs.astral.sh/uv/) и Python ≥ 3.12.

```bash
# Синхронизация окружения и установка всех зависимостей
uv sync
```

`uv` сам создаст `.venv/` и установит пакеты из `pyproject.toml` / `uv.lock`.

## Запуск

### Через uvicorn

```bash
uv run uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

### Или просто:

```bash
uv run python main.py
```

После запуска откройте в браузере **Swagger UI**:

```
http://localhost:8000/docs
```

## Endpoints

| Метод | Путь                          | Описание                                            |
|-------|-------------------------------|-----------------------------------------------------|
| GET   | `/health`                     | Проверка работы сервиса                             |
| GET   | `/terrain/info`               | Информация о загруженной матрице высот              |
| GET   | `/viewshed/vec`               | GeoJSON, **векторизация numpy** (R3 + билинейка)    |
| GET   | `/viewshed/vec/visualize`     | PNG того же                                         |
| GET   | `/viewshed/aw`                | GeoJSON, **Amanatides–Woo + numba** (быстро)        |
| GET   | `/viewshed/aw/visualize`      | PNG того же                                         |
| GET   | `/viewshed/loop`              | GeoJSON, **чистый Python-цикл** (эталон, медленно)  |
| GET   | `/viewshed/loop/visualize`    | PNG того же                                         |

### Параметры запроса (общие для всех `/viewshed/*`)

| Параметр | Тип    | Описание                                      |
|----------|--------|-----------------------------------------------|
| `x`      | float  | X-координата станции (индекс колонки)         |
| `y`      | float  | Y-координата станции (индекс строки)          |
| `h`      | float  | Высота станции над уровнем рельефа (≥ 0)      |
| `r`      | float  | Радиус обзора в ячейках (> 0)                 |

## Конфигурационный файл `config.yaml`

```yaml
heights_path: "alt.csv"         # путь к CSV с матрицей
csv_delimiter: ","              # разделитель в CSV
matrix_rows: 50                 # ожидаемое число строк (валидация)
matrix_cols: 50                 # ожидаемое число столбцов
output_dir: "output"            # куда сохранять PNG
algorithm:
  ray_step_multiplier: 2.0      # плотность шагов трассировки
```

Путь к `config.yaml` можно переопределить через переменную окружения:

```bash
VIEWSHED_CONFIG=/path/to/my_config.yaml uv run uvicorn main:app
```

## Формат матрицы высот

CSV-файл из `matrix_rows` строк и `matrix_cols` столбцов с числами —
высотами рельефа в каждой ячейке. Строка 0 — нижний край карты
(если `origin='lower'` в визуализации). Координаты в API:
`x` = индекс столбца, `y` = индекс строки.

Сгенерировать пример можно скриптом:

```bash
uv run python generate_terrain.py
```

## Что показывает визуализация

Левая панель — рельеф в цветах `terrain` + полупрозрачная зелёная
маска видимых ячеек. Красная звезда — станция, пунктир — радиус
обзора.

Правая панель — тот же рельеф (полупрозрачный) и поверх него —
зелёные точки `MultiPoint`-геометрии: по точке в каждом видимом
узле сетки. «Дыры» от перекрывающего рельефа видны как пустые
места между точками.

## Алгоритмические замечания

- Сложность: O(N² · R), где N — сторона матрицы, R — радиус.
  Для 50×50 расчёт занимает доли секунды.
- Высота станции `h` добавляется к высоте ячейки рельефа в точке
  установки — то есть это высота **антенны над землёй**.
- Граница радиуса включается в расчёт (`dist ≤ r`).
- Для очень больших матриц алгоритм можно векторизовать или
  использовать R3-алгоритм с предвычислением углов.
- **Плотность шагов трассировки.** `viewshed_loop` подбирает число
  проб **отдельно для каждого луча**: `n_steps = max(|dx|, |dy|) ·
  ray_step_multiplier`. То есть число проб пропорционально длине
  луча — короткий луч получает мало проб, длинный много, но шаг
  пробы в клетках сетки остаётся примерно одинаковым (~0.5 клетки
  при `ray_step_multiplier = 2`). `viewshed_vec` ради broadcast'а
  numpy использует **один общий** `n_steps`, посчитанный по самому
  длинному лучу в матрице, и применяет его ко всем целям. Короткие
  лучи получают избыточное число проб (по нескольку в одной
  клетке) — на корректность это не влияет (берётся max угла), но
  позиции bilinear-сэмплов отличаются от тех, что выбрал бы
  `viewshed_loop`, поэтому на пограничных клетках возможны
  единичные расхождения между двумя реализациями.

### vec/loop vs aw — это два разных модельных подхода

Реализации `viewshed_vec` / `viewshed_loop` и `viewshed_numba` (aw)
отличаются **не только по скорости, но и принципиально — стратегией
выборки рельефа вдоль луча**. Они отвечают на немного разные вопросы.

| | `viewshed_vec` / `viewshed_loop` | `viewshed_numba` (aw) |
|---|---|---|
| Алгоритм | R3 с билинейной интерполяцией | Amanatides–Woo: обход клеток сетки |
| Где берёт высоту | в `n_steps` пробах вдоль непрерывной линии, билинейно по 4 узлам | в каждой клетке, которую луч физически пересекает, сырая высота узла |
| Эффект | рельеф сглаживается между узлами | каждая клетка на пути работает как столб полной высоты |

На **гладком рельефе** (как `alt.csv`) обе стратегии дают почти
одинаковую видимость — расхождение в единицы клеток. На
**«шумных» матрицах** (случайные значения, мелкозернистый рельеф,
много локальных пиков) расхождение становится драматическим: aw
систематически считает меньше клеток видимыми, потому что каждая
клетка на длинном луче независимо подкидывает высокий шип, и `max
угла` у aw растёт быстрее, чем у vec/loop, где соседи усредняют
пики. В литературе это известный артефакт: R3+bilinear склонен
**переоценивать** видимость на шумных DEM, а per-cell-обход — её
**недооценивать**.

Какой ответ «правильный» — зависит от интерпретации матрицы высот.
Если она представляет точечные замеры на узлах, aw ближе к делу.
Если это сэмплы непрерывной поверхности — vec/loop.
